# Методика оценки и метрики

## Корзина
17 обезличенных сценариев `test-basket/TB-01..TB-17` (15 обязательных task4 §5 + 2 доп.:
TB-04 глубокое извлечение, TB-10 деградация с пустым трекером). Эталон `expected/*.json`
для TB-01..TB-12 сгенерирован из вывода rule-движка (`scripts/gen_expected.py`) после
ручной сверки; для TB-13..TB-17 зафиксирован **вручную до прогонов** (`manual_expected=True`,
task4 §20) — `gen_expected.py` их пропускает.

## Сопоставление
- **Поручение** = совпадение по (action, owner, due, source) с нормализацией
  (нижний регистр, схлопывание пробелов). Частичное совпадение — FP+FN (строгая
  метрика), отмечается отдельно.
- **Решение** = совпадение по (text, source).
- **Блокер** = совпадение по description.
- micro-агрегация: P = TP/(TP+FP), R = TP/(TP+FN), F1 = 2PR/(P+R) по суммарным TP/FP/FN.

## Метрики (task2 §12)
precision/recall/F1 поручений и решений; полнота владельцев; полнота сроков;
точность блокеров; доля утверждений с источниками (evidence coverage); время
(mean/p50/p95); доля успешных прогонов; доля прогонов с fallback; количество
ошибок; стоимость LLM; доля принятых черновиков.

success; A·P/R/F1 = 100% на замороженной корзине (детерминированный baseline). Среднее время
< 0.01 с/сценарий. Mock-LLM путь — те же метрики (прокси rule-baseline).

## Что доказывают метрики
Корзина синтетическая и выровнена с детерминированным извлечением → метрики
доказывают работу конвейера (контекст → сводка → извлечение → память → HITL →
версионирование), Safety Layer, деградацию без падения, конфликты и эволюцию навыка.
**Recall на реальных расшифровках** — замер пилота (нужна реальная LLM и боевые данные);
в MVP путь LLM-движка упражняется через mock (JSON-валидация, изоляция, ретраи).

## Движки
- `--engine rule` — офлайн baseline (метрики отчёта).
- `--engine llm --mock` — путь LLM-движка через детерминированный mock (без сети).
- `--engine llm` (с ключом) — реальная LLM; метрики требуют API-ключа.

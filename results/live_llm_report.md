## Методика семантического скоринга (LLM-judge)

- **Скрипт**: `tests/score_semantic.py` — работает поверх уже сохранённых выходов
  прогона (`run.json` каждого TB против `test-basket/TB-NN/expected/`), **без
  повторного прогона агента**.
- **Judge-модель**: `openai/gpt-4o-mini` через тот же шлюз (OpenRouter),
  temperature=0 — **не** та же модель, что оценивалась (нет self-grading).
- **Рубрика**: одно и то же действие/решение по смыслу (парафразы = совпадение);
  владелец совпадает по роли; срок совпадает по дате (или оба отсутствуют);
  source совпадает, если оба указывают на один факт (ID задачи и расшифровка,
  где он обсуждался, — один источник).
- **Протокол**: judge выносит вердикт строго-JSON `{"match": true/false, "reason"}`
  по каждой паре (извлечено × эталон) внутри кейса — 40 пар; затем жадное
  паросочетание по матрице вердиктов → TP/FP/FN → micro P/R/F1.
- **Аудируемость**: каждый вердикт (пара, вердикт, обоснование, модель judge,
  токены, стоимость) сохранён в
  `results/runs/eval_glm52_20260720/semantic_judge/pass1/` и `pass2/`;
  агрегаты — `summary.json` в каждом проходе.
- **Стабильность**: пайплайн прогнан дважды (pass1, pass2) при temperature=0 —
  **40/40 вердиктов совпали** (`semantic_judge/stability.json`), метрики
  идентичны в обоих проходах.
- **Стоимость**: pass1 $0.0037 + pass2 $0.0037 = **$0.0074** (бюджет ≤ $1).

## Оставшиеся ошибки извлечения (семантический матчинг)

1. **TB-06 (FN, поручения)**: GLM 5.2 вернула `actions: []` — поручение

2. **TB-04 (FP, поручения)**: извлечено лишнее «зафиксировать блокер по OPS-77»
   (SRE) — в эталоне такого поручения нет.
3. **TB-08 (FP+FN, решения)**: извлечено «OPS-77 больше не нужен, ППРБ-адаптер
   уже задеплоен» (source: расшифровка) против эталонного «OPS-77 больше не
   нужен» (source: OPS-77). Judge в обоих проходах счёл источники разными —
   пограничный вердикт; вручную не переопределялся (цифры не подгонялись).

## Почему строгие метрики LLM ниже rule-engine

1. **Строгое текстовое сопоставление.** Харнесс сравнивает `action/owner/due/source`
   по точной нормализованной строке. LLM парафразирует: «подготовить release-notes»
   vs эталонное «подготовлю release-notes» — семантически верно, но не засчитывается.
   Семантический матчинг закрывает именно этот разрыв: 18.2% → 90.9% по поручениям.
2. **Пустые actions.** GLM 5.2 иногда возвращает `actions: []` для коротких реплик,
   где rule-engine находит поручение по паттерну (TB-06) — это реальный пропуск,
   он остаётся ошибкой и при семантическом матчинге.
3. **Source не совпадает.** LLM ставит `source: "расшифровка синка 03.07"`, эталон —
   `source: "APP-412"`. Семантическая рубрика считает их одним источником, если
   указывают на один факт.

## Баг, обнаруженный при live LLM прогоне

`_parse_json_reply` в `src/athanor/llm.py` использовал `rfind("}")` для поиска
конца JSON. LLM иногда добавляет лишнюю `}` (например `{...}}`), что давало
`JSONDecodeError: Extra data`. Исправлено через `json.JSONDecoder().raw_decode()` —
парсится первый валидный объект, хвост игнорируется. Примечание v2: в v1 отчёта
фикс был описан, но в код не попал (потерян при пересборке рабочего дерева) —
внесён повторно вместе с v2; 126 тестов проходят.

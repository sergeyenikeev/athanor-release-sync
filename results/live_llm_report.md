## Почему метрики LLM ниже rule-engine

1. **Строгое текстовое сопоставление.** Харнесс сравнивает поля `action/owner/due/source` по точной нормализованной строке. LLM парафразирует: «подготовить release-notes» vs ожидаемое «подготовлю release-notes» — семантически верно, но не засчитывается.

2. **Пустые actions.** gpt-4o-mini часто возвращает `actions:[]` для коротких реплик, где rule-engine находит поручение по паттерну «я подготовлю … до …».

3. **Source не совпадает.** LLM ставит `source: "расшифровка синка 03.07"`, эталон — `source: "APP-412"` (ID задачи). Семантически оба верны, но matching по 4 полям требует точного совпадения.

## Баг, обнаруженный и исправленный при live LLM прогоне

`_parse_json_reply` в `src/athanor/llm.py` использовал `rfind("}")` для поиска конца JSON. LLM иногда добавляет лишнюю `}` (e.g., `{...}}`), что давало `JSONDecodeError: Extra data`. Исправлено через `json.JSONDecoder().raw_decode()` — парсит первый валидный объект, игнорируя хвост. 126 тестов проходят.
